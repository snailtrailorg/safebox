## 二、重构原则

### 2.1 安全优先（不变）

### 2.2 架构驱动（不变）

### 2.3 用户体验不降级

- 登录两步流程保持
- 注册一次完成（补 idempotency_key）
- **恢复码路径禁用验证码（防死锁），采用一次性提交 + 24h 冷却期 + 加速通道 + 冻结回滚机制**（initiate 写正式=新密码+rollback_*=旧密码副本；冷却期账户锁定；freeze 回滚 rollback_*）
- **密码强制 12+ 字符，含大写/小写/数字/特符，排除弱模式**
- **敏感操作（改密、注销）需邮箱/手机验证码**
- 管理员后台 UI 作为邮箱/手机不可用时的最后恢复通道（后续规划）

### 2.4 可交付 > 学术完美（不变）

### 2.5 限流分层原则（v1.4 新增）

- **Nginx 层**：防 DDoS 粗筛，单 IP 总连接数/秒限流。不区分端点，不感知业务。
- **应用层 L1**：邮箱/手机号粒度指数退避（Redis 持久化）。防暴力破解。
- **应用层 L2**：IP 滑动窗口（Redis 持久化）。防分布式高频扫号。
- Nginx 与应用层限流互相不可替代，必须两层并存。

## 四、风险登记

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 条目数据迁移导致数据丢失 | 低 | **高** | 双格式共存，旧数据不动；encryption_version 显式标记 |
| 前端不兼容导致生产故障 | **中** | **高** | **Feature Flag 发布策略**：先部署支持双读写的代码（开关关），稳定后开开关。回滚只需关闭开关，不回滚数据 |
| 重构期间功能回归 | 中 | 中 | 每步修改后构建 + curl 验证 |
| 老用户 KDF 降级失败无法登录 | 低 | **高** | `kdf_settings IS NULL` → fallback 到 600K |
| 部署回滚后新数据不可读 | **低→中** | **高** | **Feature Flag 发布**，确保回滚时不回滚数据（旧代码不识别的列允许 NULL） |
| 全局限流缺失 | 低 | 中 | Nginx 作为 DDoS 粗筛（单 IP 总连接数）；应用层 L1（邮箱）+ L2（IP，Redis 滑动窗口）作为业务安全底线。两者并存，不可替代 |
| PW 120K 主线程阻塞 | 高 | 中 | Web Worker + 服务端下发迭代数 |

## 五、决策日志

| 日期 | 决策 | 理由 |
|------|------|------|
| 2026-07-06 | **恢复 Item Key 层** | 为未来条目共享预留扩展通道，避免 V3 全量迁移 |
| 2026-07-06 | 回退 XChaCha20 到 AES-GCM | Web Crypto API 原生支持 |
| 2026-07-06 | 维持登录两步流程 | salt 随机性导致无法一步完成 |
| 2026-07-06 | **密码强制 ≥12 字符 + 大小写/数字/特符 + 排除弱模式** | OWASP 2025 标准；安全不能让步 |
| 2026-07-06 | **恢复码路径禁用邮箱/手机验证码，改为一次性提交新密码 + 24h 冷却期 + 加速通道 + 冻结回滚** | 防止用户收不到邮件时永久锁死；一次性完成所有操作，冻结不覆盖旧数据 |
| 2026-07-06 | **改密/注销等敏感操作增加邮箱/手机验证码** | 统一敏感操作保护策略 |
| 2026-07-06 | 注册加 idempotency_key | 网络中断后重试不卡死 |
| 2026-07-06 | 后端 security/ 包名维持 | 拆包成本 > 收益 |
| 2026-07-06 | rate_limit/middleware.py 维持 | 适配器模式 |
| 2026-07-06 | **恢复应用层限流（L1 邮箱 + L2 IP），Nginx 仅做 DDoS 粗筛** | 之前错误认为 Nginx 全局限流可替代应用层限流，但 Nginx `limit_req_zone` 基于共享内存，重启/扩容丢失，多实例隔离，无法实现 Redis 持久化的业务级限流 |
| 2026-07-06 | **引入 Feature Flag 发布策略** | 避免回滚时新数据不可读，实现零数据风险回滚 |
